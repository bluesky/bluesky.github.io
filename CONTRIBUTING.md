# Contributing to the Published Documentation

This repository contains the rendered documentation published at
`bluesky.github.io`. Most files here are generated output, not the source files
that should normally be edited by hand.

## Why Built Files Are Here

This repository is the GitHub Pages publishing target for the Bluesky Project
documentation. GitHub Pages serves static files directly. It does not run Sphinx
when someone visits the site, so the Sphinx source files must be built into
HTML, CSS, JavaScript, images, search indexes, and source-view text files before
they can be published.

The upstream project repositories own the documentation source. This repository
stores the rendered output from those projects so URLs under `bluesky.github.io`
can be served directly. For example:

```text
bluesky/bluesky
  docs/tutorial.rst          # source file

bluesky.github.io
  bluesky/tutorial.html      # rendered page
  bluesky/_sources/*.txt     # Sphinx source-view files
  bluesky/_static/*          # Sphinx assets
  bluesky/searchindex.js     # Sphinx search data
```

This publishing model keeps the website simple to host, but it means manual
edits in this repository are fragile. The next docs deployment can overwrite
changes that were not made in the upstream source repository.

## Edit the Source Repository

For project documentation changes, edit the upstream source repository for the
project you are changing. For the Bluesky tutorial, use:

```bash
git clone https://github.com/bluesky/bluesky.git
cd bluesky
git switch -c update-bluesky-tutorial
```

Edit the source file:

```text
docs/tutorial.rst
```

Do not use `bluesky/_sources/tutorial.rst.txt` in this repository as the source
of truth. It is generated by Sphinx from `docs/tutorial.rst`.

## Build the Bluesky Docs

Install the development dependencies in the `bluesky/bluesky` checkout:

```bash
python -m pip install -e ".[dev]"
```

Build the documentation:

```bash
tox -e docs
```

The upstream CI uses this same docs command. It runs Sphinx roughly as:

```bash
sphinx-build -EW --keep-going -T docs build/html
```

After the build, the generated files are under:

```text
build/html/
```

For the tutorial, the generated files include:

```text
build/html/tutorial.html
build/html/_sources/tutorial.rst.txt
```

## Preview Locally

From the upstream `bluesky/bluesky` checkout:

```bash
cd build/html
python -m http.server 8000
```

Then open:

```text
http://localhost:8000/tutorial.html
```

## Updating This Repository Manually

In normal development, prefer opening a pull request against the upstream source
repository, such as `bluesky/bluesky`. The docs publishing workflow will generate
and publish the rendered files.

If this repository must be updated manually, copy the relevant generated files
from the upstream `build/html/` directory into the matching project directory
here. For the Bluesky tutorial, keep these files consistent:

```text
bluesky/tutorial.html
bluesky/_sources/tutorial.rst.txt
```

Avoid hand-editing generated HTML and `_sources/*.txt` files independently. They
can drift, and the next Sphinx build will overwrite manual edits.
